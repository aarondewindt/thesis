The purpose of this example is to gain a better understanding of the basic workings of \acrlong{rl} and how the learning process is affected by the hyperparameters $\alpha$ and $\gamma$. The problem, based on Junell et al. \cite{junell2015reinforcement}, consists of a quadrotor agent within a grid world containing three collapsed buildings and an upload station. The goal of the agent is to visit the disaster sites to collect pictures and every once in a while upload its memory contents at the upload station.

Each time the quadrotor visits a disaster site its memory state is increased by one and the agent is given a reward of 8. In total the agent has the memory capacity of 5. Whenever the agent visits the upload station it is given a reward equal to the square of the memory state and the memory state is reset back to zero. After each visit to either a disaster site or the upload station, the agent is moved to a random location in the grid world. If the agent leaves the grid world it's given a negative reward and is moved back to its last position on the grid. As possible actions the agent can choose to move to any adjacent grid location including diagonal movements.

The agents use a simple \eps-greedy on-policy learning method using the update rule shown in Equation~\ref{fig:qr_learning_rule}. Figure~\ref{fig:qr_mean_rewards} shows the mean rewards of $1000$ agents with varying hyperparameters $\alpha$ and $\gamma$. As it can be seen decreasing the learning rate $\alpha$ reduces the convergence rate, but results in better learned policies in the long run.

Having $\gamma=0$ results in poor learning performance, this is because each update is only dependent only on the reward the agent received after performing the action. It doesn't take future action-states into account. In contrast when setting $\gamma$ to a non-zero value, the update is dependent on the value of the following action-state. Thus giving the agent an indication of the value of future events.

\begin{equation}
    Q(S_t, A_t) \leftarrow Q(S_t, A_t) + \alpha \left [  R_{t+1} + \gamma Q(S_{t+1}, A_{t+1})- Q(S_t, A_t) \right ]
    \label{fig:qr_learning_rule}
\end{equation}

\end{figure}

Figure~\ref{fig:qr_learned_policy} shows the greedy policy of one of the agents at each memory state after the learning process. Being an \eps-greedy on-policy learning method it can be seen that the agents avoid traveling along the edges of the grid world, so as to avoid stepping outside during a random action.

It can also been seen that at low memory the agents prefer to travel to the disaster sites and as the memory fills up it becomes more valuable to the agents to travel to the upload station. This is thanks to the increase in rewards the agents are able to get from the upload station as the memory fills up. Once full, the only place where the agents are able to get a reward is from the upload station, so in this case the policy always guides the agent towards the upload station.

\section{Example: Suborbital flight}
The goals of this example are to develop the simulation and to experiment with the learning methods and libraries that will be used throughout the main thesis project. 

\subsection{Simulator}

The simulator is implemented in Python using Numba to optimize computation time. It takes place in a 2D world where the vehicle has 3 degrees of freedom, the $x$ and $y$ position and a pitch angle $\theta$. A non-rotating analog of the moon is chosen as the celestial body at which to simulate. The main reasons for this choice are:

\begin{enumerate}
	\item A smaller celestial body means shorter distances and thus shorter simulation times are needed to ascent to orbit. Thus the learning process should go faster.
	\item There already are real world designs of single stage to lunar orbit spacecraft that can be used as examples during early learning runs.
	\item No atmosphere.
\end{enumerate}

The reference frames used in the simulation are shown in Figure~\ref{fig:to_refframes}. The inertial frame, $I$, is located at the center of the moon.  The vehicle-carried normal lunar reference frame, $E$, has it's origin at the surface of the moon right under the location of the vehicle, with the y-axis being normal to the surface. The body frame, $B$, has its origin at the vehicle center of mass and is oriented with the vehicle structure. Furthermore it's assumed to be in vacuum, thus no aerodynamic forces nor moments are included. Later simulations may include an atmosphere.

The set of differential equations are shown in Equation~\ref{eq:to_eom}. These are solved using a third order \emph{Adams–Bashforth} (AB3) integrator. This is an explicit, multi-step integration method that can approximate a third order solution using the results from previous integration steps. Since the previous steps need to be known, a forth order \emph{Runge–Kutta} integrator is used for the first four iterations. The main advantage of using an Adams–Bashforth integrator is that the derivatives only need to be evaluated once per integration step while still yielding higher order results, thus improving computational efficiency while minimizing integration errors.

\begin{align}
	\dot{x^I_x} &= v^I_x & \dot{v^I_x} &= a^I_x = \frac{F^I_{T_x}}{m} \nonumber \\ 
	\dot{x^I_y} &= v^I_y & \dot{v^I_y} &= a^I_y = \frac{F^I_{T_y}}{m} \label{eq:to_eom} \\
	\dot{m} &= \frac{|F_T|}{gI_{sp}} & \dot{\theta^I} & \quad \text{(input)} \nonumber		
\end{align}

The simulation supports multi-stage launch vehicles. By default the main control input is the pitch angle rate, $\dot{\theta^I}$, but the simulation includes configurable PID controllers for controlling the pitch and flight path angles.\newline

\subsection{The environment}
The vehicle used during this example is a small single stage vehicle incapable of reaching orbit. Instead the goal given to the agent is to maximize the apocynthion during a suborbital flight. The vehicle properties are shown in Table~\ref{tab:to_vehprop}.

		$m_{prop}$ & $0.2$ \si{kg} \\
		$I_{sp}$  & $80$ \si{s} \\
		$F_T$ & $6.8$ \si{N}		
	\end{tabular}
	\caption{Suborbital launch vehicle properties.}
	\label{tab:to_vehprop}

In this environment the simulation is configured to end if the vehicle contacts the lunar surface or at engine burnout. The reward is only given at the terminal state and is defined by the following equation.

\begin{equation}
	R_{t_{end}} = \left[ \frac{h + \frac{v^{E2}_y}{2g}}{4600} \right]^{21}
\end{equation}

The numerator of the outer division is the vehicle altitude plus the altitude it would be able to ascent with the remaining vertical kinetic energy. This gives an estimate of the apocynthion the vehicle can reach with its current state. This estimate is then divided by the maximum apocynthion the vehicle can potentially reach in ideal conditions to get a normalized reward. The reward is then raised to the power of $21$ in order to lower the rewards given at low apocynthions, thus encouraging the agent to manoeuvre to higher apocynthions.

The only action available to the agent is a continues pitch rate command between $-1$ and $1$ \si{rad.s^-1}. The observation is a tuple containing $\left(\gamma^E, \theta^E, \frac{h}{1200} \right)$. The altitude is normalized by dividing by $1200$ \si{rad.s^-1}.

The reason why the reward and altitude are normalized is because machine learning libraries tend be optimized or their default hyperparameters are chosen with the assumption that the inputs and outputs are normalized.

Furthermore the vehicle's initial position is always on the north pole and the initial velocity is always zero. The initial pitch angle, $\theta^E$ is randomly selected between $[15\deg,165\deg)$.

\subsection{The agent}
The agent is implemented using RLlib (Liang et al. \cite{liang2017rllib}), a \acrlong{rl} Python library designed to take advantage of parallel computing. It provides a set of primitive abstractions around the different \acrshort{rl} components, thus allowing the use of different machine learning frameworks, learning algorithms, environment, etc. RLlib also contains implementations for common \acrshort{rl} algorithms which will be used as a basis during this thesis.

An actor-critic agent is used in this example. The model consists of a \acrfull{ffnn} with two hidden layers, each containing $32$ neurons. All layers are shared between the policy and value functions and finally the agent is trained using a \acrshort{ppo} trainer following the methods described in Section~\ref{sec:ppo_algo}.

As already mentioned RLlib allows the use of parallel computing to accelerate the training process. For the PPO trainer the setup consists of a central learner process and a set a worker processes that run episodes using the latest model while the central learner is updating the model. After the central learner is done updating the models, the workers are updated with the new model and then the simulation results are send to the central learner to repeat the learning process. 

This allows the agent to learn from a varied set of episodes during each model update, thus increasing the number of visited states and actions, and prevent one "bad" episode to significantly influence the update.

\subsection{PID controller}
To have something to compare to a controller was developed consisting of a PID outer loop controlling the flight path angle, $\gamma^E$, and sending a reference signal to a PID inner loop controller controlling the pitch angle, $\theta^E$, by sending pitch rate commands to the environment. To maximize the apocynthion the controller is given a constant flight path angle reference of $\gamma^E=90^\circ$.

\subsection{Results}
The training processes lasted for $3$ hours, $50$ minutes and $31$ seconds, and ended when the model started outputting $NaN$ values. In total the agent was trained for $2252$ batches, each with on average $12.1$ episodes. Figure~\ref{fig:to_rewards} shows the average reward in each batch and the rewards achieved by agents following the greedy policy after each batch at different initial pitch angles $\theta^E$. The figure also shows the rewards achieved by the PID controller.

Figure~\ref{fig:to_losses} shows the KL-divergence, policy loss and value function loss during each update. As explained in Section~\ref{sec:ppo_algo} the KL-divergence is a measure of difference between two probability distributions, in this case the policies before and after training. The policy and value function losses are the terms corresponding to the policy and value function errors in the objective function being optimized. Together these three values give an indication for when the agent is learning and updating its model.

	\centering

From the results it is clear that most of the learning happens within the first $500$ batches, after which it remains mostly stagnant. The agent is able to reach and occasionally exceed the reward achieved by the PID controller.

Just after $900$ batches, the greedy agent reward dips down and the value function losses increase. This may be a sign of overfitting, which would also explain why the model returned $NaN$ values on it's last episode.


\subsection{Conclusion}
During the development of this experiment is became clear that the main challenges are the tuning of hyperparameters, reward function selection and limited computational resources. Thanks to the use of \acrshort{ppo} the number of hyperparameters requiring tuning is minimal and it is mainly the model structure that needs to be tuned. 

Selecting an appropriate reward function is important. For both the problem in this example and the main thesis the performance of a specific episode is only known at the end of each episode. This makes it difficult for the agent to learn, since it only receives a positive reward if it achieves the desired end state by chance.

In this example this issue was solved by giving the agent a small but positive reward to guide it towards the goal. Once close to the goal, the reward is increased by a few orders of magnitude, thus teaching the agent its main goal. This is a form of \emph{reward shaping}, where the agent is trained by learning intermediate goals leading towards the main goal using prior knowledge of the environment.

This solution may not neccesarally be appropriate for the main thesis problem since this prior knowledge is not available when it comes to orbital ascent trajectories, thus a different solution for the sparse reward is required.
